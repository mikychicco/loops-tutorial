Loops_

You keep retyping "check it again". /loop types it for you, on a schedule, while you do something else. This page takes you from zero to your first running loop, with a simulator so you can see one fire before you trust it with real work.

01

The problem it kills

Some work is not one prompt. It is the same prompt, over and over: is the deploy done yet? did CI pass? are the tests green now? Without loops you are the scheduler. You sit there, re-asking, or you walk away and the session goes idle.

/loop hands the re-asking to Claude Code. You describe the recurring task once. The session re-fires it on an interval, does the work each time, and reports back. It keeps running while the session is open, and it knows how to stop.

One sentence to remember A loop is a prompt that re-sends itself until you press Esc, the task is provably done, or seven days pass.
02

Four shapes, one command

Everything about /loop reduces to two questions: did you give it an interval? and did you give it a prompt? Two yes/no answers, four shapes.

/loop 5m check the deploy interval + prompt

The workhorse. Your prompt re-runs every 5 minutes, like clockwork. Intervals take s, m, h, d, and odd values get rounded to the nearest clean cron step (Claude tells you what it picked).

/loop fix failing tests until green prompt only · self-paced

The smart one. No interval, so Claude picks the delay itself (1 minute to 1 hour), waiting less while things are moving and more when they are quiet. It prints each chosen delay with a reason, and it can end the loop on its own once the task is provably complete.

/loop 15m interval only · built-in prompt

The janitor. No prompt, so a built-in maintenance prompt runs every 15 minutes: continue unfinished work, tend the current branch's PR (reviews, failed CI, merge conflicts), then light cleanup. It does not start new initiatives.

/loop bare · self-paced janitor

Hands-off mode. Built-in maintenance prompt, self-chosen pace. You can replace the built-in prompt by writing your own to .claude/loop.md (project) or ~/.claude/loop.md (global). Edits take effect on the next iteration, even mid-loop.

Bonus shape: loop another command

The prompt can be a slash command, so any skill you already use becomes repeatable:

/loop 20m /review-pr 1234

    05

    How loops end

    Nothing here runs forever. Four exits, in the order you will meet them:

    You press Esc

    While a loop waits for its next run, Esc clears the pending wakeup. It never fires again. This only works for loops started with /loop syntax.

    It finishes itself

    Self-paced loops (the no-interval kind) can stop on their own once the task is provably complete: tests green, deploy live, PR merged.

    You cancel by name

    Ask in plain language: "what scheduled tasks do I have?" then "cancel the deploy check". Each task has an 8-character ID. A session holds up to 50.

    7-day expiry

    Every recurring task self-deletes 7 days after creation: one final run, then gone. A forgotten loop cannot haunt you. Resuming a session within 7 days restores unexpired loops.

    The catch Loops live inside the session. Close the terminal, lose the SSH connection, and the loop dies with it. For automation that must survive your laptop, use /schedule (cloud routines) instead.
    06

    Loop, goal, or schedule?

    Three commands keep Claude working without you. They answer different questions:

    CommandFires whenLivesReach for it when
    /loop A time interval elapses (fixed or self-paced) In your session, on your machine, max 7 days "Keep checking X while I work on Y." Polling, monitoring, babysitting.
    /goal After every turn, until a condition holds In your session "Don't stop until tests pass." A target, not a timer.
    /schedule A cron schedule, on Anthropic's cloud Independent of your machine, survives restarts "Every morning at 9, triage new issues." Durable, unattended automation.

    Rule of thumb: timer → loop, target → goal, calendar → schedule.

    Recipes to steal

    Deploy watch. Fixed 2-minute poll; reports the moment it flips.

    /loop 2m check the deploy, is it finished? if so tell me the status

    Tests until green. Self-paced, so it can end itself when the suite passes. The best first loop to try.

    /loop check if all tests pass, fix any failures and re-run

    PR babysitter. Half-hourly sweep of CI, conflicts, and review comments.

    /loop 30m check CI on my open PR, rebase if conflicted, address new review comments

    Project default. Drop this in .claude/loop.md and a bare /loop becomes your project's own janitor (max 25,000 bytes, keep it tight):

    Check the release/next PR. If CI is red, pull the failing job log, diagnose, and push a minimal fix. If new review comments arrived, address each one and resolve the thread. If everything is green and quiet, say so in one line.
    08

    What bites beginners

    Tokens compound

    Every iteration re-sends the whole conversation: history, system prompt, every log you ever read. Long loops over big contexts get expensive quietly. Check /usage early and often, run /clear before starting an unrelated loop, and consider Sonnet for dumb polling: it is plenty smart for "is the build done" and much cheaper than Opus or Fable.

    Watch the first two iterations

    Never start a loop and walk away immediately. Watch it fire once or twice, confirm it is doing what you meant, then leave. A loop that misunderstood you compounds its misunderstanding on schedule.

    Give it something checkable

    "Improve the codebase" loops forever because nothing proves it is done. "Run the tests, fix failures, stop when green" has a verdict Claude can run. Self-paced loops especially need a verifiable finish line, since they decide for themselves when to stop.

    Timing is approximate, on purpose

    Fixed intervals get jitter (up to 30 minutes or half the interval) so thousands of loops don't fire at once. If a scheduled tick passes while Claude is busy, it fires once when idle, not once per missed tick. Loops are for monitoring, not for cron-precision triggers.

    Permissions carry over

    A loop inherits the session's permission mode. A loop in bypass mode pushes, deletes, and deploys without asking, every iteration, unattended. For your first loops, stay in a mode where risky actions still prompt.

    Cloud providers On Bedrock, Vertex AI, and Microsoft Foundry, self-paced mode falls back to fixed 10-minute intervals, and bare /loop just prints usage. The fixed-interval shapes work everywhere.

    10

    Your first loop, today

    1. Pick a task you re-ask anyway.

      A deploy you keep checking, a flaky test suite, a PR waiting on CI. Loops earn their cost on tasks you were already repeating by hand.

    2. Start self-paced with a checkable finish.

      Run the "tests until green" recipe above. Self-paced loops can end themselves, so your first loop also teaches you how loops stop.

    3. Watch iteration one and two.

      Confirm the loop understood you. Note the delay it picks and the reason it prints.

    4. Practice the exits.

      Press Esc during a wait. Ask "what scheduled tasks do I have?". Cancel one by name. Now nothing about loops can surprise you.

    5. Check /usage after an hour.

      See what the loop actually cost. That number tells you whether to loop with Fable, or hand the polling to Sonnet.
